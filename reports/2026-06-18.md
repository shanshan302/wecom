# 企业微信 AI / Agent 情报日报 - 2026-06-18

## 今日结论

暂无重大变化。

截至 2026-06-18（Asia/Shanghai）公开可验证信息显示，企业微信 / WeCom 仍未出现“面向普通企业，允许 Agent 或 API 以成员身份更广泛主动发送消息到任意对象”的官方新增开放。更明确的信号仍是：

- WeComTeam 正把消息、文档、智能表格、会议、待办等能力包装成 `wecom-cli` 与 OpenClaw 插件，供 AI Agent / MCP 风格工具调用，但仍带有明确入口、身份、规模和权限边界。
- `wecom-openclaw-plugin` 明确写出主动消息、群聊、MCP 集成、定时投递等能力，这说明“Agent 驱动的企业微信消息触达”在插件层已经成熟；但其发送主体仍更接近机器人 / 自建应用 / 受控通道，不等同于官方全面开放“员工代发”。
- 2026-06-17 新开的 `wecom-cli` issue 继续集中在初始化、Bot ID/Secret、URL 回调模式和消息授权状态暴露，说明当前开发者主要卡在“接入与权限可见性”，而不是官方刚刚新增了更大的消息权限。

## AI / Agent / MCP / 智能机器人动态

- [官方][访问日期 2026-06-18] `wecom-openclaw-plugin` README 已把 MCP 写进正式能力表，并将 `wecom_mcp` 与媒体、SmartPage 创建/导出拦截器并列。这比“社区自建 MCP Server”更强，因为它来自 WeComTeam 官方仓库。来源：[wecom-openclaw-plugin README](https://github.com/WecomTeam/wecom-openclaw-plugin)
- [官方][2026-06-17] `wecom-cli` open issues 中新增 `feat(auth): 希望暴露「消息」MCP 授权有效期和到期状态`。这不是新能力发布，但说明消息类 MCP/授权态已经真实进入开发者使用面，且权限可观测性成了新痛点。来源：[wecom-cli issues](https://github.com/WecomTeam/wecom-cli/issues)
- [官方][2026-06-17] 同日另有 issue 询问“url回调模式下的机器人能使用这个cli吗？”以及“init 需要支持指定 botid、secret 参数初始化”。这说明当前生态正在从单一交互式初始化，转向更适合自动化/批量部署/回调模式的接入诉求。来源：[wecom-cli issues](https://github.com/WecomTeam/wecom-cli/issues)
- [半官方/生态][访问日期 2026-06-18] 腾讯云 ADP 产品页仍在强调“智能体开发平台”定位，案例覆盖保险、证券、物流、制造、教育等行业，多数是企业内 AI 助手 / 多智能体应用 / 知识问答 / 流程自动化。今天没有直接看到“企业微信连接器新增能力”的文案，因此只能把它视为腾讯云企业 Agent 底座持续扩张，而非 WeCom 通道放宽。来源：[腾讯云 ADP](https://cloud.tencent.com/product/adp)
- [半官方/生态][2026-06-03] 腾讯轻量云 OpenClaw 更新日志页面仍可见最近修改时间为 2026-06-03，但本次抓取未展开到具体条目；能确认的是腾讯云仍持续维护 OpenClaw 产品化更新日志。来源：[轻量云 OpenClaw 功能更新日志](https://cloud.tencent.com/developer/article/2628942)

## 消息发送与 API 开放迹象

- [官方][访问日期 2026-06-18] 当前可确认的“发送主体”仍主要是三类：`wecom-cli` 的 API 模式智能机器人、`wecom-openclaw-plugin` 的 Bot / Agent 通道、自建应用调用官方消息接口。已确认的是“受控发送能力存在”；未确认的是“更广泛的成员身份代发”已开放。来源：[wecom-cli README](https://github.com/WecomTeam/wecom-cli)、[wecom-openclaw-plugin README](https://github.com/WecomTeam/wecom-openclaw-plugin)
- [官方][访问日期 2026-06-18] `wecom-openclaw-plugin` 明写可主动发给 users / groups / departments / tags，并支持 Cron 定时投递；这意味着发送模式已经从被动回复扩展到“计划任务 + 主动出站”。但 README 也写明 Cron 走 `Agent outbound channel`，要求配置 `corpId` / `corpSecret` / `agentId`，因此更像“应用侧受控外发”，不是员工账号任意代发。来源：[wecom-openclaw-plugin README](https://github.com/WecomTeam/wecom-openclaw-plugin)
- [官方][2026-06-17] `wecom-cli` issue 新增“消息 MCP 授权有效期和到期状态”诉求，说明权限模型里至少存在可失效的消息授权，不是无边界永久权限。这对后续判断开放程度很关键。来源：[wecom-cli issues](https://github.com/WecomTeam/wecom-cli/issues)
- [官方][2026-06-09 至 2026-06-17] `wecom-cli` issues 中仍持续出现三类需求：解除 10 人限制、员工服务 CLI、群公告（内部群 + 外部群）。这些都是“开发者想要更宽消息/服务权限”的直接证据，但目前都还是需求或待产品决策，不是已开放能力。来源：[Issue #82](https://github.com/WecomTeam/wecom-cli/issues/82)、[wecom-cli issues](https://github.com/WecomTeam/wecom-cli/issues)
- [社区][2025-11-13，低可信] V2EX 讨论依然反映典型现实：读消息“官方 API 可做”，但“往外面发就不行”，开发者往往在 RPA 与逆向之间权衡；还有用户声称“单个群一个月只能发 4 条”。这些都不是官方文档，不能作为能力事实，但能说明私域/外部群主动消息需求依旧未被官方通用方案充分满足。来源：[V2EX 讨论 1](https://www.v2ex.com/t/1172486)、[V2EX 讨论 2](https://www.v2ex.com/t/1171617)

明确区分：

- 已确认能力：机器人/自建应用/插件通道下的受控消息发送、群聊处理、主动消息、定时投递、MCP/Skill 化调用。
- 仍属推断或未确认：API 是否可进一步放宽到外部联系人、客户群、员工身份任意主动触达；群公告、员工服务 CLI、10 人以上消息 CLI 是否会扩容。
